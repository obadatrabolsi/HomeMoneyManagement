# مدير الأموال الشخصي — مواصفة تصميم المرحلة الأولى (MVP)

> التاريخ: 2026-06-29
> النطاق: المرحلة الأولى فقط (MVP). المراحل اللاحقة لها دورات (مواصفة → خطة → تنفيذ) مستقلة.

## 1. نظرة عامة

تطبيق إدارة أموال شخصي **offline-first**، عربي الواجهة (RTL)، يعمل كـ **PWA** دون اتصال أو تسجيل دخول. كل البيانات محلية في IndexedDB.

### نطاق الـ MVP
- إدارة الحسابات المتعددة (CRUD + أرشفة).
- العمليات: دخل، مصروف، تحويل.
- التصنيفات (رئيسية/فرعية، دخل/مصروف).
- لوحة التحكم.
- البحث والتصفية.
- النسخ الاحتياطي والاستعادة (JSON).

### خارج نطاق الـ MVP (مؤجّل لمراحل لاحقة)
الميزانيات، الأهداف، العمليات المتكررة، الديون، التقارير المتقدمة، الإحصائيات، القوالب، المرفقات، تعدد العملات بالتحويل/أسعار الصرف، تصدير SQLite، التشفير، المزامنة، تغليف Capacitor الأصلي، الإشعارات.

## 2. القرارات المعتمدة

| القرار | الاختيار |
| --- | --- |
| لغة/اتجاه الواجهة | عربي فقط (RTL) |
| العملات | عملة لكل حساب (تخزين فقط، بلا تحويل/إجمالي موحّد) |
| تنظيم المشروع | دورات حسب المراحل؛ هذه المواصفة للمرحلة الأولى |
| طبقة البيانات/الحالة | مستودعات فوق Dexie + `useLiveQuery`؛ Zustand لحالة الواجهة |
| المنصة | PWA أولًا؛ تأجيل Capacitor |
| النسخ الاحتياطي | JSON فقط؛ تأجيل SQLite |

## 3. التقنيات

React 18 + TypeScript + Vite + Tailwind CSS (RTL) · Dexie.js (IndexedDB) + `dexie-react-hooks` · Zustand · React Hook Form + Zod · date-fns · Recharts (مخطط دائري للوحة التحكم) · Vitest + `fake-indexeddb` للاختبار.

## 4. نموذج البيانات

المبالغ تُخزّن كـ **أعداد صحيحة بأصغر وحدة (سنتات)** لتفادي أخطاء الفاصلة العائمة. المعرّفات `uuid` نصية.

### `accounts`
`id` · `name` · `description?` · `icon` · `color` · `currency` (ISO، مثل `EUR`) · `initialBalance` (سنتات) · `isArchived` (bool) · `sortOrder` · `createdAt` · `updatedAt`

> الرصيد الحالي **محسوب** = `initialBalance` + مجموع عمليات الحساب (الإيداعات موجبة، السحوبات سالبة)، وليس حقلًا مخزّنًا.

### `categories`
`id` · `name` · `type` (`income` | `expense`) · `icon` · `color` · `parentId?` · `sortOrder` · `isArchived`

> التحويلات بلا تصنيف.

### `transactions`
`id` · `type` (`income` | `expense` | `transfer`) · `amount` (سنتات، موجب دائمًا) · `accountId` · `categoryId?` · `date` (ISO) · `time?` · `notes?` · `tags` (string[]) · `merchant?` · `transferGroupId?` · `counterAccountId?` · `createdAt` · `updatedAt` · `deletedAt?`

اتجاه التأثير على الرصيد يُشتقّ من `type` وطرف التحويل (سحب/إيداع).

### `settings` (صف واحد)
`defaultCurrency` · `theme` (`light` | `dark` | `system`) · `schemaVersion`

### الفهارس (Dexie)
- `transactions`: `accountId`, `categoryId`, `date`, `type`, `transferGroupId`, `deletedAt`, ومركّبة `[accountId+date]`.
- `accounts`: `isArchived`, `sortOrder`.
- `categories`: `type`, `parentId`, `isArchived`.

## 5. البنية (feature-based)

```
src/
  db/            مخطط Dexie + المستودعات (accountsRepo, transactionsRepo, categoriesRepo, settingsRepo, backupRepo)
  features/
    accounts/    قائمة، تفاصيل، نموذج إضافة/تعديل، أرشفة
    transactions/ القائمة، نموذج العملية، إجراءات السحب، التراجع
    categories/  إدارة التصنيفات (رئيسية/فرعية)
    dashboard/   ملخصات + مخطط دائري
    backup/      تصدير/استيراد JSON
  components/ui/ Button, Sheet, Modal, Toast, Fab, EmptyState, ...
  stores/        Zustand: filters, theme, selectedAccount
  lib/           formatters (عملة/تاريخ), uuid, money (سنتات), zod schemas
  i18n/          نصوص عربية
  routes/        مسارات React Router
```

كل وحدة لها مسؤولية واحدة وواجهة واضحة وقابلة للاختبار باستقلال.

## 6. تدفّق التحويل

`transactionsRepo.createTransfer({ fromAccountId, toAccountId, amount, date, ... })` يكتب **عمليتين** ضمن **معاملة Dexie ذرّية واحدة**:
- سحب من `fromAccountId`،
- إيداع في `toAccountId`،
- تشتركان في `transferGroupId`، وكل طرف يشير إلى الآخر عبر `counterAccountId`.

التعديل/الحذف يطبّق على المجموعة كاملة. **قيد الـ MVP:** التحويل بين حسابين **بنفس العملة فقط** (لا تحويل عملات).

## 7. الحذف والتراجع

الحذف = ضبط `deletedAt` (حذف ناعم). الاستعلامات تستبعد العمليات المحذوفة. يظهر Toast بزر "تراجع" لعدة ثوانٍ يعيد `deletedAt = null`. حذف طرف من تحويل يحذف المجموعة كاملة (ويستعيدها التراجع كاملة). تنظيف نهائي للعمليات المحذوفة قديمًا يُجرى عند النسخ الاحتياطي/التهيئة.

إجراءات السحب على العملية: تعديل · حذف · تكرار · إضافة للمفضلة.

## 8. لوحة التحكم

- إجمالي الرصيد **مجمّعًا حسب العملة** (سطر لكل عملة، دون جمع عملات مختلفة).
- رصيد كل حساب.
- دخل/مصروف اليوم.
- ملخص الشهر (دخل، مصروف، صافٍ).
- آخر العمليات.
- مخطط دائري لتوزيع مصروفات الشهر حسب التصنيف (Recharts).
- FAB للإضافة السريعة: مصروف · دخل · تحويل.

## 9. البحث والتصفية

- بحث نصّي: الملاحظات، التاجر، الوسوم، المبلغ.
- فلاتر: الفترة الزمنية، النوع، الحساب، التصنيف، الوسوم.
- ترتيب: التاريخ/المبلغ تصاعدي/تنازلي.
- منفّذ عبر فهارس Dexie واستعلامات `useLiveQuery` التفاعلية.

## 10. النسخ الاحتياطي والاستعادة

- **تصدير:** ملف JSON يحوي كل الجداول + `schemaVersion` + طابع زمني.
- **استيراد:** التحقق من التوافق ثم **استبدال كامل مع تأكيد** (الأبسط والأأمن للـ MVP).
- التصميم يترك مجالًا لإضافة الدمج وصيغة SQLite لاحقًا.

## 11. المظهر والتنقّل

- ثيم فاتح/داكن/اتباع النظام (Tailwind + متغيّر في `settings`).
- تصميم محمول أولًا، RTL كامل.
- React Router: `/` (لوحة التحكم) · `/accounts` · `/accounts/:id` · `/transactions` · `/categories` · `/settings`.

## 12. استراتيجية الاختبار

- **وحدة (Vitest + fake-indexeddb):** المستودعات، حساب الأرصدة، التحويل الذرّي، الحذف/التراجع، أدوات المال (سنتات) وتنسيق العملة.
- **مكوّنات:** نماذج العمليات وتحقّق Zod، المسارات الحرجة.
- **TDD** أثناء التنفيذ: اختبار فاشل أولًا ثم التنفيذ.

## 13. معايير القبول (MVP)

1. إنشاء/تعديل/أرشفة حسابات بعملات مختلفة، وعرض رصيدها المحسوب صحيحًا.
2. إضافة دخل/مصروف وانعكاسه فورًا على الرصيد ولوحة التحكم.
3. تحويل بين حسابين بنفس العملة ينشئ عمليتين مرتبطتين ذرّيًا.
4. حذف عملية مع إمكانية التراجع خلال المهلة.
5. تصنيفات رئيسية/فرعية قابلة للإدارة، وربط العمليات بها.
6. بحث وتصفية يعملان على فهارس Dexie.
7. تصدير واستيراد JSON يستعيد الحالة كاملةً.
8. عمل كامل دون اتصال (PWA)، دون تسجيل دخول.
