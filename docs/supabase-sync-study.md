## 1. الخلاصة التنفيذية (Verdict)

**الربط مع Supabase ممكن ومناسب جدًا لهذا التطبيق، وبتكلفة/مخاطرة معقولة.** البنية الحالية شبه مثالية لإضافة المزامنة لأن:

- **كل المعرّفات (IDs) هي UUID v4** (`crypto.randomUUID()` في [uuid.ts](../src/lib/uuid.ts)) — أي أن السجلات المُنشأة دون اتصال لن تتصادم عند رفعها للسحابة. هذه أكبر عقبة في أنظمة المزامنة، وهي محلولة أصلًا.
- **المبالغ مخزّنة كأعداد صحيحة (cents)** — لا مشاكل دقة عائمة عند النقل.
- **التطبيق أصلًا offline‑first** (Dexie/IndexedDB + PWA + service worker) — لا نغيّر الفلسفة، بل نضيف طبقة مزامنة فوقها.
- **طبقة بيانات مركزية نظيفة** في [src/db/](../src/db/) (repositories) — كل الكتابات تمرّ من مكان واحد، فنقطة الحقن للمزامنة واضحة.

**التوصية باختصار:** نبني **محرّك مزامنة مخصّص خفيف** (Dexie ⇄ Supabase عبر `@supabase/supabase-js`) بدل تبنّي إطار مزامنة ثقيل (RxDB / PowerSync / ElectricSQL) الذي سيفرض إعادة كتابة طبقة التخزين بالكامل. تسجيل الدخول **إلزامي**: التطبيق مقفول بالكامل حتى يسجّل المستخدم دخوله بحساب Supabase، وبعدها تُحفظ الجلسة ويبقى مفتوحًا (بما فيه العمل offline) حتى تسجيل الخروج الصريح.

### 🆕 متطلبات محدّثة (طلب المستخدم)

1. **قفل إلزامي بالكامل حتى تسجيل الدخول:** لا يُستخدَم التطبيق في أي حالة قبل الدخول. بعد الدخول تُحفظ حالته (persistent session) ويظل مفتوحًا عبر إعادة التشغيل والعمل دون اتصال (§5).
2. **واجهة حالة المزامنة:** شاشة تعرض كل العناصر غير المُزامَنة (المعلّقة) وتُحدّث حالتها إلى «تمّت المزامنة» فور اكتمالها (§8‑bis).
3. **الحساب الافتراضي:** `obada.trabolsi@gmail.com` (بريد Gmail حقيقي يتيح التحقق وإعادة التعيين) بكلمة مرور تُضبط في لوحة Supabase.

أهم فجوتان يجب معالجتهما قبل المزامنة (تفاصيل في §6):
1. **الحذف الصلب (hard delete)** في معظم الجداول — يجب تحويله إلى **tombstones** (`deletedAt`) وإلا لن تنتشر عمليات الحذف بين الأجهزة.
2. **غياب `updatedAt`** في 5 كيانات — يجب إضافته لأنه أساس حل التعارض (Last‑Write‑Wins).

## 3. لماذا Supabase؟ والبدائل

Supabase مناسب لأنه يوفّر في منصة واحدة كل ما نحتاجه:

- **Postgres**: نموذج علائقي يطابق كياناتنا مباشرة.
- **Supabase Auth**: مصادقة email/password جاهزة (لحساب obada).
- **Row Level Security (RLS)**: عزل بيانات كل مستخدم على مستوى قاعدة البيانات (`user_id = auth.uid()`) — أمان قوي دون كتابة backend.
- **Supabase Storage**: لتخزين صور المرفقات (بدل الـ Blobs في IndexedDB).
- **Realtime** (اختياري لاحقًا): دفع التغييرات فوريًا بين الأجهزة.
- **طبقة مجانية سخيّة** وقابل للاستضافة الذاتية.

**لماذا محرّك مخصّص وليس إطار جاهز؟**

| الخيار | الحكم |
|---|---|
| **محرّك مخصّص (Dexie ⇄ supabase-js)** ✅ **موصى به** | يبقي كل كود Dexie و`useLiveQuery` الحالي كما هو. نضيف فقط: حقول مزامنة + جدول outbox + خدمة sync. جهد متوسط، تحكّم كامل. |
| RxDB + Supabase replication | محرّك مزامنة ناضج، لكنه يستبدل طبقة التخزين (رغم وجود Dexie adapter، تتغيّر واجهة القراءة/الكتابة كليًا). إعادة كتابة كبيرة. |
| PowerSync / ElectricSQL | ممتازة، لكنها تفرض عميل SQLite محلي خاص بها بدل IndexedDB → إعادة كتابة طبقة البيانات بالكامل ومكوّن إضافي في البنية. |
| Dexie Cloud | الأنسب تقنيًا لأن التطبيق Dexie أصلًا، **لكنه خدمة Dexie وليس Supabase** — يخالف طلبك الصريح بـ Supabase. |

الخلاصة: الحالة الحالية (UUIDs + cents + soft‑delete جزئي) تجعل المحرّك المخصّص قليل المخاطر مقارنةً بكلفة تبنّي إطار ثقيل.

## 4. البنية المقترحة (Offline‑First + Sync)

المبدأ: **IndexedDB يبقى مصدر الحقيقة المحلي، والواجهة تقرأ منه فقط (لا تتغيّر).** المزامنة عملية خلفية ثنائية الاتجاه.

```
        الكتابة المحلية (كما هي اليوم)
   UI ──► repositories (src/db/*Repo.ts) ──► Dexie/IndexedDB
                                               │  (كل كتابة تُعلّم السجل dirty + تُسجَّل في outbox)
                                               ▼
                                        ┌──────────────┐
                                        │  Sync Engine │  (جديد)
                                        └──────┬───────┘
                    push (outbox → upsert)     │     pull (updated_at > cursor)
                                               ▼
                                   Supabase (Postgres + RLS + Auth + Storage)
```

- **القراءة:** تبقى عبر `useLiveQuery` من Dexie — صفر تغيير في المكوّنات الـ21.
- **الكتابة:** تبقى فورية على IndexedDB (يعمل offline دائمًا)، ويُضاف فقط وسم dirty/outbox.
- **المزامنة تُشغَّل عند:** إقلاع التطبيق (إن كان online)، حدث `online` (عودة الاتصال)، فترة زمنية دورية، وبعد الكتابة المحلية (debounced). لاحقًا: Supabase Realtime للدفع الفوري.
- **بوابة الدخول إلزامية:** التطبيق مقفول حتى الدخول (§5). بعد أول دخول ناجح تُحفظ الجلسة محليًا، فيعمل التطبيق كاملًا offline على القراءة والكتابة، وتُستأنف المزامنة تلقائيًا عند عودة الشبكة. أي **«الوضع العام offline»** يتحقّق بعد الدخول الأول، لا قبله.

## 5. المصادقة: قفل إلزامي + جلسة دائمة

> **متطلب المستخدم:** التطبيق مقفول بالكامل في كل الحالات حتى تسجيل الدخول؛ وبعد الدخول يحتفظ بحالته.

### 5.1 الحساب
- ننشئ مستخدمًا في **Supabase Auth** بالبريد `obada.trabolsi@gmail.com` وكلمة مرور تُضبط في لوحة Supabase (لا تُحفظ في المستودع).
  - ⚠️ ملاحظة أمنية: الكلمة ضعيفة نسبيًا وشاركتَها نصًّا؛ يُنصح بتغييرها وتفعيل إعادة التعيين عبر البريد (متاح لأنه Gmail حقيقي).

### 5.2 بوابة الدخول الإلزامية (`AuthGate`)
نضيف `AuthGate` يلفّ التطبيق **قبل** أي شيء (يحلّ محلّ سلوك «لا PIN = مفتوح» الحالي). المنطق:

```
عند الإقلاع:
  اقرأ الجلسة المحفوظة (supabase.auth.getSession)
  ├─ توجد جلسة صالحة (أو refresh token) → افتح التطبيق (offline أو online)
  └─ لا توجد جلسة → اعرض شاشة تسجيل الدخول واحجب كل شيء
```

- **حالة `loading`** أثناء قراءة الجلسة (شاشة فارغة/سبينر)، ثم `authed` أو `login`.
- لا يوجد وصول لأي صفحة (Dashboard/Transactions/…) قبل `authed` — نضع `AuthGate` أعلى `HashRouter` في [App.tsx](../src/App.tsx)، بنفس نمط `LockGate` الحالي.

### 5.3 الجلسة الدائمة + العمل offline (النقطة الحرجة)
- `supabase-js` يخزّن الجلسة (JWT + refresh token) في `localStorage` مع `persistSession: true` و`autoRefreshToken: true`، فيبقى المستخدم «مسجّلًا» عبر إعادة التشغيل دون إعادة إدخال كلمة المرور.
- **أول تسجيل دخول يتطلّب اتصالًا** (للتحقق من Supabase). بعده تعمل كل العمليات offline اعتمادًا على الجلسة المحفوظة.
- **سياسة انتهاء الجلسة offline (موصى بها):** نعتبر **وجود جلسة محفوظة على الجهاز كافيًا لفتح التطبيق offline** حتى لو لم نستطع تجديد التوكِن (لعدم وجود شبكة). نحاول التجديد بصمت عند عودة الاتصال. هكذا لا يُحبَس المستخدم خارج بياناته المحلية بسبب انقطاع الشبكة. عند عودة الاتصال إذا رفض الخادم التجديد (كلمة مرور غُيّرت/سُحبت الجلسة) → نعيد لشاشة الدخول.
- **تسجيل الخروج الصريح** فقط هو ما يقفل التطبيق ويعيد لشاشة الدخول. (قرار: هل يمسح الخروج البيانات المحلية؟ **لا** افتراضيًا — تبقى للدخول التالي بنفس الحساب؛ يمكن إضافة «خروج + مسح» لاحقًا للأجهزة المشتركة.)

### 5.4 العلاقة مع قفل PIN الحالي
- بوابة الدخول (السحابية) هي **البوابة الرئيسية الإلزامية**.
- يبقى قفل PIN الحالي **اختياريًا** كإعادة قفل سريعة محلية بعد الدخول (مثلًا عند فتح التطبيق يوميًا دون كتابة كلمة المرور كاملة). لا نرسل `pinSalt/pinHash` إلى السحابة إطلاقًا.
- الترتيب عند الإقلاع: `AuthGate` (جلسة) → ثم `LockGate` (PIN إن كان مُفعّلًا).

### 5.5 الشاشات الجديدة
- **شاشة تسجيل الدخول** (بريد + كلمة مرور + «نسيت كلمة المرور»).
- **قسم الحساب في الإعدادات** (البريد الحالي + زر تسجيل خروج + زر «مزامنة الآن» + حالة آخر مزامنة).

## 6. التغييرات المطلوبة على نموذج البيانات (Dexie v7)

نضيف **migration رقم 7** في [schema.ts](../src/db/schema.ts) (النمط الحالي يكرّر تعريف كل الجداول في كل نسخة). التغييرات:

### 6.1 حقول مزامنة موحّدة على كل الكيانات المُزامَنة
لكل صف نضيف (منطقيًا؛ بعضها موجود):
- `updatedAt: string` — **إضافته للكيانات الناقصة**: Category, GoalContribution, DebtPayment (وSettings يبقى محليًا). backfill للسجلات القديمة بقيمة `createdAt` أو `now`.
- `deletedAt?: string` — **tombstone موحّد لكل الجداول** (موجود فقط في Transaction اليوم).
- `userId?: string` — يُملأ عند تفعيل المزامنة (هوية مالك الصف).
- `syncState: 'pending' | 'synced'` (افتراضي `'pending'` عند أي كتابة محلية) — **يغذّي واجهة حالة المزامنة (§8‑bis)**. يُضبط إلى `'synced'` فور نجاح دفع الصف. اختياريًا `'error'` عند فشل متكرر. أي كتابة/تعديل/حذف محلي يعيده إلى `'pending'`.

### 6.2 تحويل الحذف الصلب إلى soft‑delete
اليوم `deleteAccount`, `deleteDebt`, `deleteCategory`… تحذف فعليًا. يجب أن تضع `deletedAt = now` بدل `db.table.delete(id)`، وأن تستثني الاستعلامات السجلات المحذوفة (كما يفعل `transactionsRepo` أصلًا عبر `.filter(t => !t.deletedAt)`). بدون ذلك، حذفٌ على جهاز لن ينتشر — بل قد «يعود» السجل من الجهاز الآخر.
- لاحقًا: مهمة تنظيف (GC) تحذف tombstones الأقدم من N يومًا **بعد** التأكد من مزامنتها.

### 6.3 جدول ميتاداتا المزامنة (جديد في Dexie)
```ts
// جدول syncMeta: مؤشّر آخر سحب لكل جدول
interface SyncMeta { table: string; lastPulledAt: string }  // key = table

// جدول outbox: التغييرات المحلية المعلّقة (بديل بسيط لوسم dirty)
interface OutboxEntry {
  id: string            // uuid للإدخال
  table: string
  rowId: string
  op: 'upsert' | 'delete'
  updatedAt: string     // ختم التغيير
}
```
> يمكن الاستغناء عن outbox المنفصل والاكتفاء بدفع كل صف `updatedAt > lastPushedAt`، لكن outbox أدقّ ويتحمّل الأعطال.

### 6.4 نقاط الحقن
بما أن كل الكتابات تمرّ من `src/db/*Repo.ts`، نضيف wrapper رفيعًا (أو hook في كل repo) يقيّد الإدخال في outbox ويحدّث `updatedAt`. `createTransfer` يجب أن يقيّد **الصفّين** كوحدة واحدة.

## 7. مخطط Supabase (Postgres + RLS)

نعكس الجداول بأسماء snake_case، مفاتيح UUID نصية، وسياسات RLS تعزل كل مستخدم. مثال (transactions؛ البقية بنفس النمط):

```sql
create table transactions (
  id                text primary key,           -- نفس UUID العميل
  user_id           uuid not null references auth.users(id) default auth.uid(),
  type              text not null,
  amount            bigint not null,            -- cents
  account_id        text not null,
  category_id       text,
  date              date not null,
  time              text,
  notes             text,
  tags              text[] not null default '{}',
  merchant          text,
  is_favorite       boolean,
  transfer_group_id text,
  counter_account_id text,
  transfer_direction text,
  created_at        timestamptz not null,
  updated_at        timestamptz not null,       -- أساس LWW
  deleted_at        timestamptz                 -- tombstone
);

create index on transactions (user_id, updated_at);   -- لكفاءة السحب

alter table transactions enable row level security;
create policy "own rows" on transactions
  for all using (user_id = auth.uid()) with check (user_id = auth.uid());
```

- كل جدول يحصل على `user_id + updated_at + deleted_at` وفهرس `(user_id, updated_at)` وسياسة RLS مماثلة.
- **Settings لا يُعكَس كما هو** (id ثابت + يحوي PIN). إن أردنا مزامنة التفضيلات، نُنشئ جدول `user_settings` بمفتاح `user_id` **بدون** حقول PIN.
- Amounts تبقى `bigint` (cents).
- يُنصح بإنشاء المخطط عبر ملف **migration SQL** في المستودع (مجلد `supabase/migrations`) لإعادة الإنشاء والتتبّع.

## 8. محرّك المزامنة (البروتوكول)

مزامنة **Last‑Write‑Wins بالاعتماد على `updatedAt`** (مناسبة تمامًا لسيناريو مستخدم واحد على عدّة أجهزة — وهو الواقع هنا). لتفادي انحراف الساعات نعتمد **ختم وقت الخادم** كمرجع للسحب.

**Pull (لكل جدول):**
```sql
select * from <table>
where user_id = auth.uid() and updated_at > :lastPulledAt
order by updated_at asc;
```
ثم نطبّق على Dexie بقاعدة LWW: نكتب الصف القادم فقط إذا كان `remote.updatedAt >= local.updatedAt`. نحدّث `syncMeta.lastPulledAt` بأحدث ختم استلمناه. الصفوف ذات `deleted_at` تُطبَّق كحذف محلي.

**Push:**
```ts
// اقرأ outbox، جمّع حسب الجدول، ثم:
await supabase.from(table).upsert(rows, { onConflict: 'id' })
// عند النجاح: احذف إدخالات outbox المقابلة
```
- ترتيب الدفع يحترم التبعية (accounts/categories قبل transactions) لأن FK موجودة في Postgres. أو نجعل FK مؤجّلة/غير مُلزِمة لتبسيط الرفع.
- **التحويلات (transfers):** ندفع صفّي المجموعة معًا؛ إن فشل أحدهما نُبقيهما في outbox لإعادة المحاولة (شبه‑ذرّية عمليًا مع LWW).

**التشغيل والجدولة:**
- عند الإقلاع (online + مسجّل) → pull ثم push.
- مستمع `window.online` → مزامنة عند عودة الشبكة.
- بعد كتابة محلية → push مؤجّل (debounce ~2‑5s).
- دوري كل بضع دقائق كشبكة أمان.
- **المرحلة 2:** Supabase Realtime channel لإشعار الأجهزة الأخرى فورًا.

**التعارض:** LWW يكفي هنا. الحقول الحسّاسة (رصيد الحساب) محسوبة من المعاملات وليست مخزّنة، فلا «رصيد» متعارض — فقط معاملات فردية، ونادرًا ما يُعدَّل نفس السجل من جهازين في آنٍ.

## 8‑bis. واجهة حالة المزامنة (Sync Status UI)

> **متطلب المستخدم:** واجهة تعرض كل العناصر التي لم تُزامَن بعد، وتُحوّل حالتها إلى «تمّت المزامنة» عند الانتهاء.

### الفكرة
لكل صف حقل `syncState` (§6.1)، وواجهة تقرأ المعلّق منه تفاعليًا. بما أن القراءة عبر `useLiveQuery`، **تتحدّث الشاشة تلقائيًا** لحظة انتقال الصف من `pending` إلى `synced` دون أي كود تحديث يدوي.

### مصدر البيانات
```ts
// عدّاد المعلّق (شارة على أيقونة المزامنة في الشريط)
const pendingCount = useLiveQuery(() => db.outbox.count(), [])

// قائمة العناصر المعلّقة مجمّعة حسب النوع
const pending = useLiveQuery(async () => {
  const rows = await db.outbox.toArray()          // أو استعلام syncState==='pending' عبر الجداول
  return groupByTable(rows)                        // {transactions: [...], accounts: [...], ...}
}, [])
```

### الشاشة (`/sync`)
- **رأس:** حالة عامة (متصل/غير متصل)، آخر مزامنة ناجحة، زر **«مزامنة الآن»**، وعدّاد إجمالي المعلّق.
- **قائمة مجمّعة حسب النوع** (معاملات، حسابات، ديون، …) وكل عنصر يُظهر وصفًا مختصرًا + **شارة حالة**:
  - 🟠 `pending` بانتظار المزامنة · 🔵 `syncing` جارٍ · 🟢 `synced` تمّت · 🔴 `error` فشل (مع زر إعادة محاولة).
- عند اكتمال المزامنة تنتقل العناصر إلى `synced` فتختفي من قائمة «المعلّق» (أو تظهر لحظيًا بعلامة خضراء ثم تُزال) — **حيّ عبر `useLiveQuery`**.
- **شارة عامة** على أيقونة المزامنة في شريط التنقّل/الرأس تعرض `pendingCount` (تختفي عند 0).
- **إدارة الأخطاء:** عناصر `error` تبقى ظاهرة مع سبب الفشل وزر إعادة المحاولة (يعيدها إلى outbox/pending).

### أين تُدمج
تبويب/صفحة جديدة `/sync` ضمن [AppShell](../src/routes/AppShell.tsx)، مع أيقونة حاملة للشارة في الرأس. قسم الحساب وتسجيل الخروج (§5.5) يمكن وضعهما في الصفحة نفسها أو الإعدادات.

## 9. المرفقات (Attachments → Supabase Storage)

- اليوم: Blobs داخل IndexedDB، ومُستثناة من الـ backup.
- المقترح (مرحلة لاحقة لتقليل النطاق): رفع الـ Blob إلى **Supabase Storage** بمسار `userId/attachmentId`، وتخزين المسار/الميتاداتا في جدول `attachments`. تنزيل كسول عند الحاجة على جهاز آخر.

## 10. الحفاظ على البيانات الحالية للعميل (حرِج)

هذا شرطك الأول، وهو أكثر حساسية الآن لأن الدخول أصبح إلزاميًا: المستخدم الحالي لديه بيانات محليًا **دون حساب**، وأول تحديث سيطالبه بتسجيل الدخول. الخطة:

1. **نسخة احتياطية إلزامية قبل أي شيء:** أول تشغيل بعد التحديث نُجبر تصدير JSON كامل (الميزة موجودة في [backupRepo.ts](../src/db/backupRepo.ts)) كشبكة أمان قبل لمس البيانات.
2. **migration محلي (v6→v7) غير مُدمّر:** يضيف الحقول الجديدة (`updatedAt/deletedAt/userId/syncState`) دون مسّ القيم؛ backfill لـ`updatedAt` الناقص من `createdAt`، وكل الصفوف الحالية تبدأ `syncState='pending'` لترفعها المزامنة.
3. **تبنّي البيانات المحلية عند أول دخول (claim):** بما أن الدخول إلزامي، عند أول تسجيل دخول ناجح على هذا الجهاز نُسند `userId` (= معرّف مستخدم obada) لكل الصفوف المحلية الموجودة، ثم نرفعها عبر upsert (initial seed). المعرّفات UUID فالرفع آمن من التصادم. هكذا لا تضيع بيانات ما قبل الحساب.
4. **حالة وجود بيانات سحابية مسبقة:** ندمج بالـ id عبر LWW (لا استبدال مُدمّر مثل الـ import الحالي).
5. **قابلية التراجع:** المزامنة لا تحذف البيانات المحلية أبدًا؛ تسجيل الخروج (دون «مسح») يُبقي كل شيء محليًا سليمًا للدخول التالي.

> الفرق الجوهري عن الـ `importBackup` الحالي (الذي يمسح كل الجداول ثم يعيد الإضافة): محرّك المزامنة **يدمج** ولا يمسح.

## 11. المخاطر والتخفيف

| الخطر | التخفيف |
|---|---|
| انحراف ساعات الأجهزة يكسر LWW | اعتماد `updated_at` من الخادم كمرجع للسحب؛ اختياريًا `default now()` عند الكتابة السحابية |
| الحذف الصلب لا ينتشر | تحويل كل الحذف إلى tombstones (§6.2) قبل تفعيل المزامنة |
| كيانات بلا `updatedAt` | إضافته + backfill (§6.1) |
| تسرّب PIN للسحابة | استثناء `pinSalt/pinHash` صراحةً؛ Settings محلي فقط |
| صفّا التحويل ينتشران جزئيًا | دفعهما معًا + إبقاؤهما في outbox حتى نجاح الاثنين |
| خطأ في RLS يسرّب بيانات | اختبارات عزل صريحة لكل جدول قبل الإطلاق |
| ازدواج التصنيفات الافتراضية | `seedDefaultCategories` يعمل على كل إقلاع ويطابق بـ`type:name`؛ يجب ألا يُنشئ صفوفًا جديدة تُرفع كتكرارات — نجعل التصنيفات الافتراضية بمعرّفات ثابتة (deterministic) أو نمنع رفع المُولّدة تلقائيًا |
| حدود الطبقة المجانية / انقطاع الشبكة | كل شيء يعمل offline؛ المزامنة تُعيد المحاولة؛ مراقبة الحصص |
| حبس المستخدم خارج التطبيق offline بسبب الدخول الإلزامي | الجلسة المحفوظة تفتح التطبيق offline دون تجديد التوكِن؛ التجديد يُحاوَل بصمت عند عودة الشبكة (§5.3) |
| أول دخول يتطلّب اتصالًا | مقبول لمرة واحدة عند التثبيت/التحديث؛ بعده لا حاجة للشبكة للفتح |
